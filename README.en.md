# Beyond Choices / Know Me

[中文说明](README.md) · [View English](README.en.md) · [Download Chinese Markdown](README.md?raw=1) · [Download English Markdown](README.en.md?raw=1)

Beyond Choices / Know Me is an AI interview-style life direction exploration mini program for students after the Chinese national college entrance exam.

It is not a traditional college application tool. It does not provide admission score lines, admission probability, major rankings, or an “AI admissions influencer” experience. Its goal is to help students organize how they understand majors, cities, universities, real-world constraints, and future directions before making application decisions.

## Core Idea

Reality as the ground, ideals as the light.

The product does not make decisions for users. It helps users see why they choose what they choose.

## Product Principles

- Do not create anxiety.
- Do not impose standard answers.
- Do not treat employment as the only standard.
- Do not avoid real-world constraints.
- Do not preach.
- Do not write motivational fluff.
- Do not reduce every question to exam scores.
- Keep a sense of life exploration.
- Use the college entrance exam as the main context, without becoming an ordinary application tool.

## Core Flow

The current experience is built around six question cards:

1. Where am I standing now?
2. What initial ideas do I have about the future?
3. What do I care about more?
4. What worries me most?
5. When ideals and reality conflict
6. What do I hope university will bring me?

## Result Page

The result page should generate:

- A personalized summary
- Several tags that match the user's own expression
- A structured view of expectations for majors, cities, and universities
- A reminder of real-world risks
- Follow-up questions worth thinking about

Tags should not be hard-coded. They should be generated by AI based on the user's answers whenever possible. Examples include, but are not limited to:

- Ideal Exploration
- Steady Growth
- City Driven
- Interest First
- Opportunity Seeking

## Writing Style

The default language is Chinese: restrained, clear, and companionable.

The copy should not feel like a personality test, an admissions agency advertisement, or a motivational article. Every sentence should carry information and avoid vague repetition.

## Open Source Safety Notes

This repository has been cleaned up for safer open sourcing:

- API Keys are not hard-coded in source code.
- The DeepSeek API Key is read from the cloud function environment variable `DEEPSEEK_API_KEY`.
- The real WeChat mini program appid in `project.config.json` has been replaced with `touristappid`.
- `.gitignore` excludes `project.private.config.json`, `.env`, certificates/private keys, dependency folders, build caches, and the production mini program QR code.
- Real user data, cloud function secrets, database exports, and paid/business logic should not be committed.
- The open-source version should only keep demo prompts. If production prompts, business rules, or paid features are added later, keep them in private configuration or a private branch.

If a security concern cannot be fixed directly in code, exclude the sensitive file through `.gitignore` and document what future users need to provide themselves.

## What You Need To Provide

After cloning the repository, future users need to provide the following before the mini program can run:

1. Import the project into WeChat DevTools and replace `touristappid` in `project.config.json` with your own mini program appid.
2. Deploy `cloudfunctions/deepseekChat` and `cloudfunctions/chatStore` in WeChat Cloud Development.
3. Configure environment variables for the `deepseekChat` cloud function:
   - `DEEPSEEK_API_KEY`: required. Your own DeepSeek API Key.
   - `JINA_API_KEY`: optional. Used for web-search support. The basic AI interview flow can still run without it.
   - `DEEPSEEK_TIMEOUT`, `SEARCH_TIMEOUT`, `SEARCH_RESULT_LIMIT`: optional. See `.env.example` for default values.
4. Prepare a cloud database collection named `chat_sessions`. It is recommended to read and write it only through cloud functions, not as a public collection.
5. If you need share cards with a QR code, provide your own `miniprogram/assets/mini-code.jpg`. This file is ignored by default to avoid publishing a production mini program QR code to GitHub.
6. If you integrate real payment, membership, analytics, marketing, or operations configuration, keep the related logic and secrets in a private repository or deployment environment.

## Do Not Commit

- Real API Keys, `.env`, certificates, or private keys.
- `project.private.config.json`.
- Exported user chat records, openids, phone numbers, scores, rankings, or other personal data.
- Production mini program QR codes unless you explicitly want repository visitors to enter your live mini program.
- Paid logic, business configuration, production prompts, or internal operations strategy.
